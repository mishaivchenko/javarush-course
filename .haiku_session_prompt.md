# Session Prompt: Build Haiku 50 — The Right Way

## Context

We are building a **Haiku 50** web app — a Japanese minimalistic haiku generator powered by OpenAI API. This is our JavaRush homework assignment. We have:

- A **reference project** at `https://haiku-50.onrender.com` (HTML/CSS/JS frontend template, Express backend)
- A **20-slide methodology presentation** teaching evidence-based codebase analysis and structured development
- Existing project structure under `/Users/mishaivchenko/dev/javarush_course/` (multi-lesson frontend projects)

**The goal is NOT to copy the reference.** The goal is to APPLY the methodology from the presentation: audit before code, evidence before claims, spec before implementation, tasks before execution.

## Rules of Engagement

### 1. No Vibe Coding
Every claim must have an evidence reference (file:line, command output, config value). If there's no reference, it's labeled as a hypothesis. Claude is a research assistant, not an oracle.

### 2. Safe Start
Always begin with `pwd` → `git status` → `ls`. Discovery and editing are separate phases — no changes during discovery.

### 3. Artifacts Before Code
We produce documents in this order:
1. `HANDOFF_REVIEW.md` — audit of the template: what works, what only looks like it works, what's missing, with evidence
2. Design spec → `docs/superpowers/specs/YYYY-MM-DD-haiku-50-design.md`
3. Task decomposition → `docs/superpowers/specs/tasks/NNN-*.md`
4. Code, one task at a time

### 4. API Contract First
Before writing backend or frontend code, define:
```
POST /generate-haiku
request: { words: string[] (3-7), language: string (1 of 12), wasabiLevel: number (0-6) }
response: { haiku: string (exactly 3 lines) }
```
Pass `language`, `words`, `wasabiLevel` as separate fields. Never concatenate.

### 5. Prompt Architecture (Separate from Code)
Two-layer prompts stored in their own file:
- **Base layer**: 3 lines, chosen language, user's words, conciseness, poetry
- **Spiciness layer**: 0-6, from calm classic to surreal but safe
- **Safety layer**: no profanity, aggression, prohibited content
- **Format normalization**: strip markdown wrappers and intros; if not exactly 3 lines, error or retry

### 6. Validation on Both Sides
Frontend validates for UX. Backend validates as the source of truth (frontend is easily bypassed).

### 7. Error Handling as a Separate Step
Not mixed into the main flow. Walk it separately.

### 8. Security
API key in `.env` only. `.env` in `.gitignore`. No key in code, chat, or repo.

### 9. Prompt Design Leverage
For `gpt-5-nano-2025-08-07`:
- `reasoning_effort`: low or medium (cheaper, faster)
- `verbosity`: low (exactly 3 lines, no surrounding text)

### 10. Subagent for Heavy Reading
When many files need scanning but only a summary is needed — use a subagent. Define an output contract: summary, evidence (file:line), assumptions, open questions, next steps.

## Work Order

### Phase 1: HANDOFF_REVIEW
Audit the reference template as a foreign codebase. Produce `HANDOFF_REVIEW.md` with:
- File structure, UI states, handlers, localStorage
- Languages (12)
- Buttons and handlers
- Mocks/fallbacks
- History (last 100 in browser)
- Security: API key in frontend?
- Gaps with evidence

### Phase 2: Spec
Design document covering architecture, API contract, prompt design, error handling, component tree, data flow. No code.

### Phase 3: Task Decomposition
Break spec into numbered tasks (001-NNN) with acceptance criteria, dependencies, files touched.

### Phase 4: Task-by-Task Implementation
One task at a time. Each task: read plan, implement, verify, commit.
